> 文档角色：PM 预拆解稿  
> 目标读者：产品、系统策划、UI/UX、客户端、服务端、测试、运营  
> 输入资料：`output/design/武侠放置网游_详细游戏机制文档.md`、`Document/【审阅稿】武侠放置网游_完整设计方案.docx`、`output/story/武侠放置网游_世界观_节点地图_背景故事线.md`

## 0. 文档目标

本稿不是重复写一版策划案，而是把现有三份资料转化为一份可直接服务于 UI 设计与研发架构规划的“功能模块拆解文档”。

本稿重点解决三件事：

1. 把首发版本的功能边界拆清楚，明确什么是必须做、什么是可预留。
2. 把前台信息架构和核心页面域梳理出来，方便 UI 开始出 IA、线框和高保真。
3. 把服务端逻辑域、核心实体、异步任务、配置簇梳理出来，方便开发做技术方案和排期。

说明：

- 本稿以首发 MVP 的“中间扩展版”为范围基准。
- 世界观与系统规则以现有设计稿为准，本稿只做产品拆解与工程补全。
- 文中出现的“账户、邮件、消息、运营后台、配置中心”等模块，属于研发落地所需的工程底座补全，不代表新增玩法。

## 1. 首发产品抽象

### 1.1 产品一句话抽象

一款以“节点江湖”为地图骨架、以“行动队列 + 离线收益”为成长底座、以“区域市场 + 风险 PvP + 帮派政治”为中长期驱动的武侠放置网游。

### 1.2 首发体验主张

- 白天 5 到 15 分钟：收菜、调队列、看日报、查市场、处理挂单。
- 晚上 30 到 90 分钟：匪巢、远征、副本、护镖、劫镖、擂台、帮战择一或组合。
- 长线目标：通过成长、制造、跑商、社交、据点与恩怨，形成“我在一个活着的江湖里经营和冒险”的体验。

### 1.3 首发范围基线

- 地图范围：高安、低安、危险区三层结构开放；危险区首发开放 `朔砂外原` 与 `伏龙渊域外围/前哨`。
- 内容范围：主线前三幕；第四幕只留接口。
- 节点范围：首发建议约 38 个节点。
- 核心玩法范围：行动队列、成长构筑、地图路线、区域市场、制造、匪巢半挂机、远征、副本、护镖/劫镖、帮派情报、危险区税收、日报、榜单、基础商业化。

### 1.4 产品系统三层结构

#### A. 经营成长层

- 行动队列
- 离线结算
- 技能成长
- 制造与采集
- 仓储与装备
- 江湖日报

#### B. 风险流通层

- 节点地图与路线
- 区域市场
- 运输与护镖
- 匪巢与掉落
- 远征与副本
- PvP 死亡、掉落、损毁、保险

#### C. 组织政治层

- 帮派
- 情报网络
- 危险区驻地与税收
- 榜单、通缉、恩怨
- 世界事件与节点易主

这三层必须在首发形成闭环：经营成长层提供产出，风险流通层制造消耗与价差，组织政治层为深度玩家提供长期政治目标。

## 3. 前台信息架构建议

### 3.1 一级导航建议

建议首发采用“底层 6 到 8 个核心导航 + 顶层轻量快捷入口”的结构，避免把复杂系统直接堆成二十多个按钮。

推荐一级导航：

1. 首页
2. 地图
3. 行动
4. 角色
5. 仓库
6. 市场
7. 江湖
8. 帮派

顶层常驻快捷入口：

- 邮件/公告
- 日报入口
- 任务追踪
- 银两与核心资源
- 小地图热区提示
- 活动与商城

### 3.2 页面域拆分

| 页面域 | 主要页面 | 主要目标 |
| --- | --- | --- |
| 首页域 | 江湖日报首页、资源概览、挂单概览、帮派简报、快捷收取 | 承担“每日首次进入”和“收菜后看全局”的入口 |
| 地图域 | 世界地图、节点详情、路线预览、热区/战争态展示 | 承担区域理解、路径决策、风险提示 |
| 行动域 | 队列总览、行动配置、匪巢结算、离线结算、异常恢复 | 承担放置核心闭环 |
| 角色域 | 属性、技能树、领悟、装备页、心法页、行具页、构筑预设 | 承担战斗与生活构筑管理 |
| 仓库域 | 背包、仓库、战利品栏、货仓、分解/整理、耐久修理 | 承担资产管理与回收 |
| 市场域 | 区域市场、挂单、成交记录、价格趋势、合同、运输订单、保险 | 承担经济循环 |
| 江湖域 | 主线章节、支线、远征、常规副本、世界 Boss、榜单、通缉 | 承担主动玩法与中长期目标 |
| 帮派域 | 帮派主页、成员列表、岗位、情报频道、据点、税率、战报 | 承担组织协作与政治玩法 |

## 4. 功能模块详细拆解

### 4.1 账户与接入模块

- 业务目标：支撑 PC/Steam 首发的登录、角色创建、公告、邮件、系统设置与基础状态管理。
- 子模块：登录鉴权、选服/选角、角色创建、命名校验、协议与设置、邮件公告、基础通知。
- 核心页面：启动页、登录页、角色创建页、选角页、邮件公告页、设置页。
- 核心实体：`Account`、`CharacterProfile`、`Session`、`MailMessage`、`PlayerSettings`。
- 服务端域：账户鉴权、角色基础资料、命名服务、系统邮件、公告下发。
- 依赖关系：所有功能的底座；主线第一章、日报首页、队列、仓库均依赖角色基础状态。
- 首发边界：支持单角色链路即可，但数据结构要预留多角色、多区服、多平台绑定能力。

### 4.2 角色与成长模块

- 业务目标：承接玩家长期成长、流派差异和构筑管理。
- 子模块：基础属性、战斗技能、生活技能、领悟节点、心法、装备位、行具、构筑预设。
- 核心流程：学习技能 -> 提升熟练 -> 解锁领悟 -> 穿戴装备/心法/行具 -> 形成当前构筑 -> 在战斗与生产中持续验证。
- 核心页面：属性页、技能树页、领悟页、装备页、心法页、行具页、构筑保存/切换弹窗。
- 核心实体：`CharacterBuild`、`SkillTrack`、`EquipmentItem`、`KungfuLoadout`、`VehicleSlot`。
- 核心配置：技能经验曲线、领悟节点、装备品质倍率、行具类型、负重与货仓规则。
- 服务端职责：成长结算、构筑校验、战斗前快照固化、负重校验、装备损毁后的状态重算。
- UI 重点：必须同屏展示“战斗收益、生活收益、负重、行具、风险”五类信息，避免玩家只看到战力而忽略跑商与挂机能力。
- 首发边界：不做复杂职业锁定，采用“技能组合形成流派”的方案；构筑预设首发至少支持 2 套。

### 4.3 世界地图与路线模块

- 业务目标：把“节点江湖”从概念变成可交互、可感知、可判定的主界面系统。
- 子模块：世界地图、节点详情、路线预览、安全等级、节点热度、章节开放、驻防与控制态。
- 核心流程：玩家打开地图 -> 查看节点与路线 -> 读取风险和收益 -> 发起移动/运输/挂机行为 -> 落点后触发功能与结算。
- 核心页面：世界地图总览、节点详情抽屉、路线预览卡、热区/情报图层、节点战报浮层。
- 核心实体：`NodeRegion`、`RouteEdge`、`SecurityBand`、`TerritoryNode`。
- 核心配置：节点开放章节、路线耗时、风险系数、路型、税率区间、世界事件开关。
- 服务端职责：出发校验、落点判定、路线可达性判断、节点控制权结算、战争态切换。
- 客户端职责：地图表现、热区与安全等级展示、路线收益/风险可视化、最近战报与节点状态提示。
- UI 重点：地图不是装饰页面，而是运输、匪巢、帮战、据点、主线推进的统一入口。
- 首发边界：服务端必须以节点与落点为核心判定，不在中间路径表现层做真实战斗判定。

### 4.4 行动队列与离线结算模块

- 业务目标：形成“上班挂机、下班主动”的日常基本盘。
- 子模块：行动队列槽位、行动计划、离线结算、异常中断、恢复、战利品等待领取、匪巢手动下一处。
- 首发行动类型：采集、制造、炼药、烹饪、打坐、清剿匪巢、运输委托、修理/整理。
- 核心流程：配置队列 -> 执行 -> 周期结算 -> 遇到中断或等待人工确认 -> 领取或恢复。
- 核心页面：队列总览、行动配置弹窗、离线结算页、异常中断弹窗、匪巢结算弹窗。
- 核心实体：`ActionQueueSlot`、`OfflineSettlementRecord`、`InterruptReason`、`LootBundle`。
- 核心配置：槽位数量、单槽时长限制、离线上限、仓库格数、战利品栏格数、等待下一处规则。
- 服务端职责：队列调度、参数快照、离线计算、异常中断、恢复点保存、热更兼容。
- 客户端职责：队列编辑、剩余时长展示、异常原因提示、一键恢复入口、匪巢结果呈现。
- UI 重点：这一页是首发留存核心页，必须把“收益、风险、阻塞原因、下一步操作”做得非常清楚。
- 首发边界：不做无限自动连刷；匪巢必须保留“手动前往下一处”的停顿点。

### 4.5 物品、仓储与资产模块

- 业务目标：承接玩家资产，支撑制造、交易、掉落、掉落损毁、货物运输和损耗回收。
- 子模块：背包、仓库、战利品栏、货仓、品质、耐久、绑定、分解、修理、堆场待取。
- 核心流程：获取物品 -> 按规则进入背包/仓库/战利品栏/货仓 -> 使用/交易/分解/损毁 -> 形成材料回流。
- 核心页面：背包页、仓库页、战利品栏页、货仓页、分解/修理弹窗。
- 核心实体：`ItemInstance`、`InventorySlot`、`WarehouseStorage`、`CargoStorage`、`DurabilityState`。
- 核心配置：物品类型、品质、堆叠规则、绑定规则、耐久公式、分解产出表。
- 服务端职责：库存变更、耐久变更、分解结算、超载校验、掉落与损毁后的资产迁移。
- UI 重点：必须清楚区分“背包”“仓库”“战利品栏”“货仓”四种容器，避免玩家认知混乱。
- 首发边界：建议先做统一物品框架，再在上层页面按容器过滤展示，避免后期容器逻辑重复开发。

### 4.6 制造、采集与经济模块

- 业务目标：形成玩家驱动经济底座，支撑材料流、成品流、银两回收与市场价差。
- 子模块：采集、蓝图、制造、炼药、烹饪、主材槽、材料变体、NPC 回收、价格锚、银两排水口。
- 核心流程：采集/掉落获得材料 -> 进入仓储 -> 使用蓝图生产 -> 成品进入自用或市场 -> 通过税费、修理、保险、维护费回收银两。
- 核心页面：采集/制造页、蓝图页、材料详情页、分解页、成本预估卡。
- 核心实体：`BlueprintRecipe`、`MaterialSource`、`CraftJob`、`EconomicSinkRecord`。
- 核心配置：蓝图表、主材槽表、材料变体表、资源来源表、经济税率表、保险参考价规则。
- 服务端职责：制造排产、材料校验、成品属性生成、成本估值、参考价维护、银两回收统计。
- UI 重点：制造页必须能清楚解释“主材影响成品”的差异化，不然系统深度难以被用户感知。
- 首发边界：先做规则清晰的少量高价值蓝图，不宜首发铺太多无差异制造配方。

### 4.7 市场、交易与运输模块

- 业务目标：把生产链、价差和风险对抗真正串起来。
- 子模块：区域市场、挂单、成交记录、价格趋势、合同、运输订单、护镖订单、过路税、资源税、交易税。
- 核心流程：玩家在本市场圈挂单/购买 -> 形成区域价差 -> 发起运输订单 -> 产生护镖与劫镖场景 -> 在税费中形成回收。
- 核心页面：区域市场页、挂单簿、历史价格页、运输利润预估、订单详情、保险估值说明。
- 核心实体：`MarketListing`、`MarketTradeRecord`、`TransportOrder`、`TaxSnapshot`。
- 核心配置：市场圈划分、商品分类、税率、保险估值逻辑、运输订单类型与时限。
- 服务端职责：挂单撮合、成交税计算、价格中位数维护、订单超时、税费分账、异常交易监控。
- UI 重点：市场页要兼顾“普通玩家快速买卖”和“深度玩家看价差与趋势”两类诉求。
- 首发边界：市场先做区域簿与简单趋势图即可，不建议首发做过重的 K 线和深度量化界面。

### 4.8 PvE 模块

- 业务目标：为首发提供主动内容、产出驱动和惊喜感。
- 子模块：匪巢、命名精英、公共痕迹、密报、远征、单人剧情副本、3 人本、5 人本、世界 Boss。
- 核心流程：进入匪巢 -> 自动战斗 -> 掉落结算 -> 可能触发密报 -> 决定自用或交易 -> 进入更高阶远征或副本。
- 核心页面：匪巢进入页、清剿结算页、密报信笺页、远征组队页、副本入口页、世界 Boss 入口页。
- 核心实体：`BanditNestInstance`、`LootBundle`、`ExpeditionClue`、`DungeonInstance`、`BossEvent`。
- 核心配置：匪巢档位、掉落池、命名精英概率、痕迹值、密报概率、远征模板、Boss 开放时窗。
- 服务端职责：模板抽取、自动战斗、掉落判定、痕迹值生成、密报生成、副本结算。
- UI 重点：匪巢结算、稀有掉落、密报信笺三类展示必须做出“武侠悬疑感”和“开奖感”。
- 首发边界：先以匪巢 + 密报远征 + 少量副本构建梯度，避免首发副本数量过多导致内容生产压力失控。

### 4.9 PvP、死亡与风险结算模块

- 业务目标：让 PvP 成为经济消耗出口与地图政治驱动，而不是单纯的数值碾压系统。
- 子模块：擂台、低安开红、危险区自由 PvP、劫镖/护镖、掉落、损毁、保险、护命符、通缉、重复击杀保护、新手保护。
- 核心流程：观察与锁定 -> 开红/接战 -> 货物掉落 -> 装备损毁 -> 保险赔付 -> 生成通缉与保护期。
- 核心页面：开红提示、战斗结算页、掉落损毁结算页、保险页、通缉榜、保护期提示。
- 核心实体：`PvPKillResult`、`InsurancePolicy`、`WantedRecord`、`ProtectionState`。
- 核心配置：掉落率、损毁率、各区风险系数、保险费率、护命符规则、重复击杀衰减。
- 服务端职责：合法性校验、掉落与损毁结算、赔付、通缉值更新、击杀收益衰减、保护期控制。
- UI 重点：必须把“哪里会掉什么、哪里会损毁什么、怎么规避风险、怎么获得保险赔付”展示清楚，否则普通玩家会把系统感知成过度惩罚。
- 首发边界：首发应优先保证劫镖/护镖和低安/危险区死亡结算稳定，擂台只承担荣誉玩法，不承担经济主出口。

### 4.10 帮派、社交与政治经济模块

- 业务目标：让帮派成为安全感来源、情报网络载体和危险区长期目标承载体。
- 子模块：帮派创建/加入、岗位、情报频道、好友、师徒、恩怨、帮派驻地、危险区据点、税率调节、维护费、战报。
- 核心流程：加入帮派 -> 承担岗位 -> 情报上报/参与运输/参与战事 -> 帮派争夺节点 -> 调整税率与设施 -> 形成长期政治生态。
- 核心页面：帮派主页、成员与岗位页、情报频道页、帮派地图热区页、据点页、税率页、战报页。
- 核心实体：`Guild`、`GuildRole`、`GuildIntelReport`、`TerritoryNode`、`GuildWarRecord`、`GrudgeRecord`。
- 核心配置：岗位权限、情报持续时间、据点维护费、税率上下限、驻地设施效果、战争时窗。
- 服务端职责：权限校验、情报置信度计算、据点维护结算、税率变更限制、战报沉淀、恩怨生成。
- UI 重点：帮派地图、情报频道、据点税率三页是帮派系统最重要的价值可视化页面。
- 首发边界：高安总舵可先轻量化；危险区据点和税率一定要首发有闭环，否则组织政治层会失去抓手。

### 4.11 叙事、主线与任务模块

- 业务目标：通过章节解锁把系统节奏自然串联，让系统不是冷冰冰地堆出来。
- 子模块：主线章节、支线、人物关系、势力关系、章节解锁、节点开放、日报播报、世界事件。
- 核心流程：完成章节 -> 解锁系统/节点 -> 日报播报 -> 触发区域余波任务 -> 推进新玩法。
- 核心页面：章节总览页、章节详情页、任务追踪页、势力关系页、角色关系页、世界事件页。
- 核心实体：`MainChapter`、`Quest`、`NPCState`、`ChapterUnlockRule`、`DailyNewsTemplate`。
- 核心配置：章节门槛、解锁系统映射、剧情文案模板、NPC 出场条件、世界事件开关。
- 服务端职责：章节完成判定、解锁、事件播报队列、日报拼装、支线触发。
- UI 重点：主线页需要承担“系统解锁说明”的职责；用户必须知道为什么解锁、解锁了什么、下一步去哪。
- 首发边界：前三幕必须和系统解锁绑定；第四幕只保留可见不可玩的深层入口。

### 4.12 江湖日报、榜单与运营反馈模块

- 业务目标：承接每日登录反馈、世界变化反馈和身份认同反馈。
- 子模块：江湖日报、帮派简报、天下要闻、个人总结、武林榜、富甲榜、铸匠榜、通缉榜、帮会战绩榜。
- 核心流程：服务端每日汇总 -> 玩家首次登录拼装个人版 -> 用户查看总结与热点 -> 进入当日目标。
- 核心页面：日报页、榜单页、增刊快讯弹窗、个人本周总结页。
- 核心实体：`JianghuDailyEntry`、`RankingRecord`、`BattleNews`。
- 核心配置：日报模板、榜单周期、榜单规则、重大事件触发规则。
- 服务端职责：日报生成、榜单结算、事件增刊生成、个人化摘要拼装。
- UI 重点：日报是整个产品“烟火气”和“世界活着”的核心页面，需要优先定义视觉基调。
- 首发边界：榜单不宜过多；先做最能体现身份差异的五类榜单即可。

### 4.13 商业化、活动与商城模块

- 业务目标：在不卖数值的前提下建立稳定收入模型。
- 子模块：外观服装、武器/行具皮肤、舒适度功能位、活动页、礼包页、商城页。
- 核心页面：商城首页、外观详情页、活动页、行具外观页。
- 核心实体：`ShopSku`、`CosmeticAsset`、`ConvenienceEntitlement`、`EventCampaign`。
- 核心配置：SKU 表、外观分类、限时活动配置、发售节奏、权限位上限。
- 服务端职责：订单校验、权益发放、限购限制、活动开关。
- UI 重点：商城必须明确“外观/便利，不卖数值”的边界表达，降低核心用户警惕感。
- 首发边界：首发只做最基础外观和少量便利位，不建议首发把活动运营复杂化。

### 4.14 配置中心、风控与后台模块

- 业务目标：支撑持续调参与问题定位，避免 live 运营失控。
- 子模块：配置中心、掉落池配置、节点配置、蓝图配置、世界事件配置、日志追踪、风控规则、GM 后台、数据看板。
- 核心页面：内部运营后台、配置发布页、日志追查页、周平衡看板。
- 核心实体：`ConfigVersion`、`DropPoolConfig`、`WorldEventConfig`、`RiskAuditLog`、`BalanceDashboard`。
- 服务端职责：配置热更、灰度发布、日志采集、异常订单审查、刷单/骗保/脚本行为识别。
- PM 重点：这部分不直接面向玩家，但决定首发是否能稳定迭代。
- 首发边界：至少要有配置热更、关键日志、运营手工补偿、简单封禁与申诉处理能力。

## 6. 开发架构建议

### 6.1 总体原则

考虑到原始方案给出的首发周期约 6 到 9 个月、团队规模约 3 到 5 人，首发不建议直接拆成重型微服务。

推荐方案：

- 后端采用“模块化单体 + 异步任务中心 + 配置中心”。
- 关键战斗、掉落、交易、死亡结算坚持服务端权威。
- 客户端主要负责展示、交互编排和局部缓存，不持有最终判定权。
- 通过逻辑域边界保持后续可拆分能力，但首发先控制工程复杂度。

### 6.2 逻辑服务边界建议

| 逻辑域 | 主要职责 | 对外依赖 |
| --- | --- | --- |
| AccessDomain | 登录、鉴权、角色入口、邮件公告 | Account、Player |
| PlayerDomain | 玩家基础资料、角色状态、引导状态 | Growth、Narrative、Queue |
| WorldDomain | 节点、路线、开放状态、落点判定、战争态 | Guild、Narrative、Economy |
| GrowthDomain | 属性、技能、领悟、构筑、行具 | Asset、PvE、PvP |
| QueueDomain | 行动计划、离线结算、异常恢复 | Growth、World、Asset |
| AssetDomain | 物品、仓储、耐久、分解、货仓 | Growth、Economy、PvP |
| EconomyDomain | 市场、挂单、订单、运输、税费、估值 | Asset、World、Guild |
| PvEDomain | 匪巢、副本、掉落、密报、远征 | Growth、Asset、Narrative |
| PvPDomain | 开红、掉落、损毁、保险、通缉、保护期 | Asset、World、Guild |
| GuildDomain | 帮派、岗位、情报、据点、税率、战报 | World、Economy、PvP |
| NarrativeDomain | 主线、任务、节点开放、日报、世界事件 | World、Player、PvE |
| OpsDomain | 商城、活动、榜单、埋点、风控、GM 后台 | 全域 |

### 6.3 异步任务与定时任务建议

首发至少要有一套统一任务调度机制，负责：

- 行动队列结算
- 离线收益结算
- 匪巢完成后的掉落与痕迹生成
- 密报过期处理
- 市场挂单过期
- 保险参考价更新
- 每日 05:00 日报世界版汇总
- 榜单周期结算
- 据点维护费扣除
- 世界事件开关切换
- 风控扫描与异常审计

### 6.4 推荐事件总线主题

为了降低模块耦合，建议从首发开始统一事件命名：

- `ChapterUnlocked`
- `QueueStarted`
- `QueueInterrupted`
- `QueueCompleted`
- `BanditNestFinished`
- `LootClaimed`
- `MarketListingCreated`
- `MarketTradeFinished`
- `TransportOrderAccepted`
- `PvPKillResolved`
- `TerritoryOwnerChanged`
- `DailyReportGenerated`
- `GuildIntelReported`

### 6.5 存储与基础设施建议

不限定技术栈，但建议满足以下能力：

- 关系型数据库：存放角色、物品、订单、市场、帮派、配置版本。
- Redis 或等价缓存：存放会话、热区、排行榜缓存、短时情报、价格快照。
- 对象存储：存放插画、邮件资源、活动资源、战报截图等静态内容。
- 日志与监控系统：追踪掉落、赔付、市场成交、异常登录、脚本行为。

### 6.6 为什么不建议首发做微服务

- 玩法复杂度已经很高，首发真正难点在规则正确与体验稳定，而不是横向扩展。
- 市场、资产、队列、PvP 结算耦合度高，首发过早拆服务会放大一致性成本。
- 小团队需要先把“业务边界清楚”放在“物理服务拆分”之前。

建议做法：

- 代码按逻辑域分层。
- 数据库按域分表。
- 事件按域发布。
- 只有当市场、战斗或网关压力明显独立后，再从模块化单体拆出单独服务。

## 7. 研发排期映射建议

结合原审阅稿的 P0-P5 节奏，建议把功能落地顺序调整为下面的工程切片。

### P0：立项与规格冻结

- 完成首发范围冻结。
- 完成实体模型与配置簇定义。
- 完成导航 IA、页面清单、组件清单。
- 完成逻辑域与模块化单体架构设计。
- 完成关键风险评估：市场一致性、离线结算、掉落/赔付可追溯。

### P1：单人闭环原型

- 账户与角色基础链路
- 地图基础展示与节点进入
- 角色成长与装备构筑
- 行动队列与离线结算
- 仓储与物品基础
- 采集、制造、蓝图
- 基础匪巢 PvE
- 首页/日报原型

目标：实现“单人收菜 -> 制造 -> 匪巢 -> 掉落 -> 整理 -> 再排队”的闭环。

### P2：联网经济闭环

- 区域市场
- 挂单与成交
- 价格趋势与税费
- 运输订单
- 基础帮派
- 基础情报频道
- 主线前半段系统解锁

目标：实现“生产 -> 市场 -> 运输 -> 价格差 -> 日报反馈”的联网闭环。

### P3：风险与组织闭环

- 开红与低安/危险区 PvP
- 掉落、损毁、保险、通缉
- 护镖/劫镖
- 命名精英、密报、远征
- 帮派岗位、据点、税率
- 榜单与战报

目标：实现“经济产出 -> 风险对抗 -> 装备消耗 -> 组织协作 -> 节点政治”的深度闭环。

### P4：内容打磨与运营能力

- 主线前三幕完整串联
- 副本/世界 Boss 完整化
- 商城与活动
- 埋点与周平衡看板
- GM/运营后台
- 新手保护与反骚扰打磨

### P5：测试与上架准备

- 数值回归
- 经济回归
- 死亡赔付与日志审计
- 引导和日报文案打磨
- Steam 上架素材与成就配置

## 8. UI 与研发协同交付建议

为避免 UI 与开发两边各做一半再返工，建议以“页面包 + 逻辑域包”双线并行。

### 8.1 UI 交付包

第一包：

- 首页/日报
- 世界地图
- 节点详情/路线预览
- 行动队列
- 匪巢结算

第二包：

- 角色构筑
- 仓储与装备
- 区域市场
- 主线章节

第三包：

- 密报/远征
- 运输订单/护镖
- 帮派情报
- 据点税率

### 8.2 研发交付包

第一包：

- 账户、角色、物品、配置中心
- 世界域、成长域、队列域

第二包：

- 经济域、市场域、运输域
- 日报、任务、章节解锁

第三包：

- PvE 域、PvP 域、帮派域
- 榜单、战报、活动与风控

## 9. 当前待定事项清单

以下问题建议在 UI 深入和技术方案评审前先定掉，否则后续会频繁返工。

1. 首发区服模型是否采用单大区 + 分线，还是多服结构。
2. 战斗表现层颗粒度采用“战报式半演出”还是“更完整的自动战斗表现”。
3. 区域市场圈的具体数量和边界是否在首发前一次性冻结。
4. 护镖/劫镖是否允许跨帮派临时组队接单。
5. 世界 Boss 的开放窗口是固定时段还是动态事件驱动。
6. 帮战是否首发即做预约时窗，还是先做节点争夺与小规模遭遇。
7. 行具系统是否首发就支持外观与功能分离。
8. 邮件、申诉、封禁、补偿的运营后台深度做到什么程度。
9. Steam 首发是否需要预留后续移动端账户互通。
10. 反脚本的识别规则与误封申诉链路由谁负责定义。
